User Accounts and Passwords

Who will administrate the application user accounts?

There is a central admin account which is able to add users, change roles and change passwords. Users currently cannot manage their own password. This most likely will be changed in the future.

We have state standards for password requirements on applications. Will we have control over setting these standards? If not, what are the application user password standards?

Currently the password scheme can be enforced by the central admin when they specify an individuals password.

Will the user be able to change their own password?

A user currently is not able to change their own password. This most likely will be changing in the future where this feature can be enabled to allow the end user to change their password. Also there would be the ablility to specifiy password requirements.

Does functionality exist to allow the users to reset forgotten passwords?

A user currently is not able to reset or change their own password. This most likely will be changing in the future where this feature can be enabled to allow the end user to change their password. Also there would be the ablility to specifiy password requirements.

What password hashing algorithm and salting process are in use?

The application uses PBKDF2 hashing algorithm. Specifically it:
  1. Creates a new 128-bit UUID. This is our salt.
  2. Creates a sha1 hash of the concatenation of the bytes of the plain-text password and the salt (sha1(password + salt)).
  3. Prefixes the result with -hashed- and appends ,salt.
More information can be found at http://docs.couchdb.org/en/2.0.0/intro/security.html

What message(s) are displayed for failed login attempts?

A failed login attempt displays the message "Invalid user name or password".

Do user ID's remain unique over time in the application?

Yes user ID's remain unique over time in the application.

Is auto-complete disabled on the user login forms?

Currently auto-complete is NOT disabled on the user login form.

Account lockout process.

Currently there is not an account lockout policy.

Inactive account policy.

Currently there is not an inactive account.

What kind of password management is associated with the app?

MMRIA leverages CouchDB 2.0 for role-based security. The 5 user roles are Abstractor, Data Analyst, Committee Member, Form Designer, and Admin. The Admin role allows for User Management. Currently we have not configured any set password requirements but our plans are to make password requirements configurable by the Admin at the state/jurisdiction level. Passwords are stored as a salted SHA1 hash.
Encryption

Does CouchDB provide encryption for the database?

CouchDB does encrypt and salt password, but it does not encrypt the database(s).

What is the encryption used for data in transit?

It is advised that a reverse proxy be setup to provided SSL certificate encryption.

How is application information relayed to the user secured?

Information relayed to user is displayed on web page and local browser data is destroyed when the user logsout.

How is data encryption handled?

It is advised that a reverse proxy be setup to provided SSL certificate encryption.
